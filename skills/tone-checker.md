# Tone Checker

## Quick Validation

Run through these checks before publishing any content:

### 1. Principles Check
- [ ] **Human** — Does it sound like how people actually speak?
- [ ] **Clear** — Are the words simple and easy to understand?
- [ ] **Inclusive** — Does it work for everyone regardless of background?
- [ ] **Helpful** — Does it have a clear purpose and help the user?
- [ ] **Engaging** — Does it add value without getting in the way?

### 2. Voice and Tone Check
- [ ] Voice is consistent with brand identity
- [ ] Tone is calibrated to context (helpful for errors/payments, personality for browsing/deals)
- [ ] Not too much personality (risks sounding inauthentic)
- [ ] Not too functional (risks sounding robotic)

### 3. Writing Quality Check
- [ ] Active voice, not passive
- [ ] Verbs lead CTAs and headings
- [ ] Sentences are short (aim for 15 words, max 25)
- [ ] Plain language throughout (Hemingway grade 6 or lower)
- [ ] No jargon, Latin phrases (eg, ie, etc), or idioms
- [ ] No ampersands (use "and")
- [ ] No unnecessary "now" in CTAs
- [ ] Sentence case used (not Title Case or BLOCK CAPITALS)
- [ ] Content is scannable (headings, bullets, bold, short paragraphs)

### 4. CTA Check
- [ ] CTAs start with a verb
- [ ] CTAs are 2-4 words
- [ ] CTAs complete the sentence "I want to..."
- [ ] CTA text matches the destination
- [ ] CTAs are descriptive (never "Click here" or "Read more")

### 5. Consistency Check
- [ ] Same terminology used throughout
- [ ] No multiple variants for the same action
- [ ] Aligned with existing content patterns and library

### 6. Accessibility Check
- [ ] Links make sense out of context
- [ ] Headings use correct hierarchy (H1 > H2 > H3)
- [ ] Headings are under 65 characters
- [ ] All images have appropriate alt text
- [ ] No images of text
- [ ] Forms have descriptive labels (no placeholder text)
- [ ] Errors are below their form field, blame-free, and solution-focused

### 7. Inclusive Language Check
- [ ] Gender-neutral pronouns used
- [ ] No exclusionary metaphors (blacklist, master, manpower)
- [ ] Person-first disability language
- [ ] No "fast/quick/easy" — use specific timeframes instead
- [ ] Diverse names in examples

### 8. Messaging Check (if applicable)
- [ ] Error: calm, specific, never blames user, provides fix
- [ ] Warning: no excessive "please"/"sorry", includes next steps
- [ ] Success: positive confirmation, no "successfully", optional delight
- [ ] Information: used sparingly, links to more detail
- [ ] Amend: includes deadline, avoids anxiety language ("hurry", "don't miss out")

### 9. Emoji Check (if applicable)
- [ ] Emojis placed after text, not before or mid-sentence
- [ ] No repeated emojis
- [ ] Emojis reinforce meaning, never replace words
- [ ] Works in both light and dark mode
